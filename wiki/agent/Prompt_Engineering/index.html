提示工程（Prompt Engineering）

提示工程是使用大模型生成文本、处理任务时，对输入提示（prompt）进行设计与优化的实践。对于指令式大模型来说，Prompt 是唯一的输入——因此好 prompt 的价值非常高，尤其在把大模型落地到产品的“最后一米”交付时，提示工程往往占据大量工作量。

为什么提示工程重要

  1. 提高准确性与可控性：好的提示可以显著减少模型“幻觉”（hallucination）和歧义回答。
  2. 降低成本：用 prompt 优化替代盲目选用更大、更贵的模型，通常能显著降低推理成本。
  3. 工程可维护：把输出格式、口径、错误处理等约束在 prompt 中，能减少后端复杂度与对输出解析的额外工作。

问：当“人人都能做 prompt”时，如何保持优势？
答：把设计 prompt 的过程工程化、可复用、并结合领域知识与外部系统来提升准确率与可验证性。

Prompt 调优（不断迭代）

寻找高质量的 prompt 是持续迭代的过程：

  • 如果知道训练数据分布，参考相似训练样本构造 prompt 更高效；
  • 如果不清楚，可参考官方示例、对比不同模型的输出，并通过试验找规律；
  • prompt 的微小改动（语序、开头/结尾、角色定义）可能导致显著差异，因此要版本化与记录每次变更的效果。

高质量 Prompt 的核心要点（具体、信息丰富、减少歧义）：

  1. 指令要具体（明确期望行为）
  2. 提供必要背景（上下文越充分越好）
  3. 明确输出格式（便于程序解析）

注意：当底层模型发生替换（或升级）时，prompt 可能需要重新验证并优化。

Prompt 的典型构成

不要盲目套模板。模板的价值是提醒我们不要遗漏要素，但不应僵化使用。典型的构成包含：

  • 角色（Role）：给模型定义适当角色（例如“你是一位软件工程师”）；
  • 指示（Instruction）：任务描述、约束与目标；
  • 上下文（Context）：相关背景信息或历史对话；
  • 例子（Examples / Few-shot）：正确或错误的示例，有助提升稳定性；
  • 输入（Input）：标注清楚任务输入内容；
  • 输出（Output）：期望的风格与格式（JSON、XML、表格等）。

与人沟通结构类似：擅长与人沟通的人，往往在 Prompt 设计上更有优势。

为什么先定义角色有效？
模型训练数据中充满了“扮演角色”的对话样本，使得开头的角色设定对模型行为有显著影响；此外，有研究发现模型对开头与结尾内容敏感（例如有关长上下文使用的研究），因此把关键信息放在开头或结尾通常更可靠。

将大模型用于系统的核心思路

graph LR A[用户] --> B[NLU: 大模型] --> C[结构化信息] --> D[传统程序处理] --> E[NLG: 大模型/模板] --> F[返回给用户]

要点：把自然语言转换为结构化信息后，用传统编程处理关键逻辑，再用模型或模板把结果转换为自然语言。

对话系统实现建议（逐步调优）

  1. 理解输入：先用简单的 prompt 验证模型是否能理解任务；
  2. 规范输出：约束输出格式（例如返回严格 JSON 或带关键字段）以便后续程序处理；
  3. 加入示例：对常见错例与正例做 few-shot，能提升稳定性；
  4. 垂直知识：为领域场景准备模板或占位变量（如 ES6 模板字符串），使输出便于替换与渲染；
  5. 增加约束：用约束控制风格、口吻与拒答策略；
  6. 统一口径：对常见的偏题或不可回答的问题提供统一回答模板，保证一致性。

常见问题与注意：

  • 多轮对话通常需要把历史上下文一并传入（会增加 token 成本）；
  • prompt 本身不会改变模型权重；
  • 传入的历史对话可能被模型提供者用于后续训练（注意隐私与合规）。

进阶技巧

思维链（Chain-of-Thought, CoT）

通过提示模型逐步拆解问题（例如以“Let’s think step by step”或“逐步思考”的指令开头），模型更倾向生成推理过程，从而提高复杂逻辑或计算题的正确率。适用场景：数学题、逻辑推理、复杂判定等。

自洽性（Self-Consistency）

为减少幻觉，可多次采样（或改变 temperature），收集多个候选解并用投票或评分机制选择最可靠结果。

思维树（Tree-of-Thought, ToT）

在每一步生成多个分支，形成“思维树”，对分支进行启发式搜索与剪枝，从而找到更高质量的解答。实现上需要设计分支生成、评估与合并策略。

Prompt 攻击与防护

常见攻击方式

  • 越狱（Jailbreak）：通过巧妙构造 prompt 让模型绕过规则；
  • 注入（Prompt Injection）：用户输入改变系统级 prompt，使得模型输出违背预期或泄露敏感信息；

防范措施

  1. 前置校验：在将用户输入传递给模型前，用 classifier 或规则检测潜在危险 prompt；
  2. 固定前缀（Preamble）：在每次用户输入前拼接固定的 system 指令，限制可执行范围；
  3. 沙箱化输出解析：把模型输出视作不可信数据，通过规则/验证器再加工；
  4. 专门检测器：结合已有的检测方案（如研究或厂商工具）来识别恶意 prompt。

注：目前尚无 100% 有效的方法，工程上应采用多层防护（检测、约束、审计）来降低风险。

提示工程实践要点（总结）

  1. 先用 prompt 试图解决问题，常常能“四两拨千金”；
  2. 但也别迷信 prompt —— 在需要高确定性或复杂业务逻辑时应结合传统编程或验证层；
  3. 定义角色、加入例子是最常用且高效的技巧；
  4. 在需要更严谨的答案时可使用思维链（CoT）；
  5. 对抗 prompt 注入与越狱需要工程化的多层防护与持续监测。

  • 执行任务（如代码、校验）用低 temperature（接近 0）；
  • 生成类创意文本可用较高 temperature（0.7–0.9）；
  • 不要盲目把 temperature 设超过 1，除非有明确需求。

用 LLM 调优 Prompt

人工 + 自动化两条路：

  • 人工调优：经验驱动，设计 role、示例、边界条件；
  • 自动化调优：例如基于遗传算法或搜索策略（如“Prompt Tune”思路）在测试集上评估并自动生成更优 prompt。

示例思路（遗传算法）：

  1. 在不改变原意的前提下，用 LLM 生成 prompt 变体；
  2. 在测试集上评估每个变体的效果；
  3. 按表现选择与交叉变异，迭代直到收敛。
